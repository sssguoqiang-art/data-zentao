# 禅道数据口径

这份文档给 Claude / Codex 判断字段含义和写只读查询时使用。目标是避免表查错、字段用错。

## 核心对象

| 对象 | 主要表 | 说明 |
|---|---|---|
| 项目 / 执行 / 版本 | `zt_project` | 同一张表承载项目、执行、sprint 版本 |
| 产品 | `zt_product` | 产品线或业务产品 |
| 产品和版本关系 | `zt_projectproduct` | `project -> zt_project.id`，`product -> zt_product.id` |
| 任务 | `zt_task` | 任务状态、负责人、工时、截止时间 |
| 需求池 / 版本需求 | `zt_pool` | 公司定制需求表，版本推进主要看这里 |
| 需求池分组 | `zt_pool_version` | 需求池桶，不是研发版本 |
| 标准需求 | `zt_story` | 标准禅道需求，不能替代 `zt_pool` |
| Bug | `zt_bug` | Bug 主表 |
| Bug 部门复盘 | `zt_bug_dept_review` | Bug 按部门记录原因和举措 |
| 待办 | `zt_to_do_list` | 当前待办和 AI 专项等 |
| 举措 | `zt_measures_management` | 举措管理 |
| 业务字典 | `zt_pool_type` | 待办状态、待办类型、问题类型等 |
| 用户 | `zt_user` | 账号和真实姓名 |
| 部门 | `zt_dept` | 部门名称和负责人 |

## 项目和版本

版本推进页面里的版本页签通常来自：

```sql
zt_project.type = 'sprint'
```

产品和版本关系：

```sql
zt_projectproduct.project = zt_project.id
zt_projectproduct.product = zt_product.id
```

按产品找当前版本的常用口径：

```sql
SELECT p.*
FROM zt_project p
JOIN zt_projectproduct pp ON pp.project = p.id
JOIN zt_product prod ON prod.id = pp.product
WHERE p.deleted = '0'
  AND p.type = 'sprint'
  AND prod.name = '平台部'
  AND p.begin <= CURDATE()
  AND p.end >= CURDATE();
```

## 需求池和版本需求

`zt_pool.pv_id` 是多态字段，不能单独解释。

| 条件 | `pv_id` 指向 | 含义 |
|---|---|---|
| `zt_pool.type=0` | `zt_project.id` | 某个研发版本下的需求 |
| `zt_pool.type=1` | `zt_pool_version.id` | 需求池分组 |

版本下需求：

```sql
SELECT *
FROM zt_pool
WHERE deleted = '0'
  AND type = 0
  AND pv_id = <版本ID>;
```

需求对应任务：

```sql
zt_pool.taskID -> zt_task.id
```

实际版本可通过任务反查：

```sql
zt_pool.taskID -> zt_task.id -> zt_task.execution -> zt_project.id
```

## 任务

常用字段：

| 字段 | 含义 |
|---|---|
| `id` | 任务 ID |
| `name` | 任务名称 |
| `project` | 所属项目 |
| `execution` | 所属执行 / sprint 版本 |
| `parent` | 父任务 ID |
| `assignedTo` | 当前负责人账号 |
| `status` | 任务状态 |
| `deadline` | 截止日期 |
| `estimate` | 预计工时 |
| `consumed` | 已耗工时 |
| `left` | 剩余工时 |
| `finishedDate` | 完成时间 |
| `delayTimes` | 延期次数 |
| `delayReason` | 延期原因 |

父子任务口径：

```text
parent > 0 且能命中 zt_task.id：子任务
parent = 0 或 parent = -1：顶层或无父级
```

当前逾期未完成：

```sql
deadline < CURDATE()
AND status NOT IN ('done', 'closed', 'cancel')
AND deleted = '0'
```

已完成但延期：

```sql
DATE(finishedDate) > deadline
AND finishedDate NOT IN ('0000-00-00 00:00:00')
AND deleted = '0'
```

## 待办

待办主表：

```text
zt_to_do_list
```

这张表的删除标记和常见禅道口径相反：

```text
deleted = '1' 表示未删除
deleted = '0' 表示已删除
```

状态来自 `zt_pool_type`：

| 状态 ID | 名称 |
|---:|---|
| 7 | 未开始 |
| 19 | 进行中 |
| 8 | 已完成 |
| 9 | 已关闭 |

未完成待办：

```sql
SELECT *
FROM zt_to_do_list
WHERE deleted = '1'
  AND status NOT IN (8, 9);
```

责任人字段：

```text
zt_to_do_list.duty_user
```

它可能是单账号，也可能是逗号分隔多账号。查询姓名时需要拆分或用 `FIND_IN_SET` 关联 `zt_user.account`。

## Bug

Bug 主表：

```text
zt_bug
```

常用字段：

| 字段 | 含义 |
|---|---|
| `id` | Bug ID |
| `title` | Bug 标题 |
| `product` | 所属产品 |
| `project` | 所属项目 |
| `execution` | 所属执行 / 版本 |
| `task` | 关联任务 |
| `assignedTo` | 当前负责人 |
| `status` | Bug 状态 |
| `severity` | 严重程度 |
| `pri` | 优先级 |
| `openedBy` | 创建人 |
| `openedDate` | 创建时间 |
| `resolvedBy` | 解决人 |
| `resolvedDate` | 解决时间 |
| `closedBy` | 关闭人 |
| `closedDate` | 关闭时间 |

Bug 部门复盘：

```text
zt_bug_dept_review.bugId -> zt_bug.id
zt_bug_dept_review.dept -> zt_dept.id
```

## 用户和部门

用户展示时：

```sql
zt_user.account -> zt_user.realname
```

部门展示时：

```sql
zt_dept.id -> zt_dept.name
```

不要直接把账号当作姓名展示给业务用户。

## 查询输出建议

回答管理分析类问题时，优先输出：

| 内容 | 说明 |
|---|---|
| 结论 | 先说数量、风险、主要集中点 |
| 口径 | 说明用了哪些表和筛选条件 |
| 明细 | 表格列出关键记录 |
| 归纳 | 按版本、部门、责任人、状态归类 |
| 疑问 | 如果字段为空或口径不稳，要明确说明 |

如果结果可能受实时数据变化影响，要说明“按当前查询时刻”。
